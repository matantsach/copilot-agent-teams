# Design: Issues #3, #4, #5, #6

## Issue #6 — Observability

**Core scope:** Duration tracking + audit log + enhanced team_status.
**Skip:** Standalone CLI, `/team-watch` skill.

### Schema changes (db.ts)

Add to `tasks` table:
- `claimed_at INTEGER` — set in `claimTask()`
- `completed_at INTEGER` — set in `updateTask()` on completion

New table:
```sql
CREATE TABLE IF NOT EXISTS agent_actions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  team_id TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  task_id INTEGER,
  action_type TEXT NOT NULL,
  detail TEXT,
  created_at INTEGER NOT NULL DEFAULT (unixepoch())
);
CREATE INDEX IF NOT EXISTS idx_actions_team ON agent_actions(team_id, created_at);
```

### New MCP tool: `get_audit_log`

Input: `{ team_id, agent_id?, limit?, action_type? }`
Returns: recent actions filtered by params.

### Changes to existing tools

- `claimTask`: set `claimed_at = unixepoch()`
- `updateTask`: set `completed_at = unixepoch()` on completion
- All tool handlers: log actions to `agent_actions` table
- `team_status`: include per-task duration and last activity per agent

## Issue #5 — Human Review Checkpoint

**Core scope:** `needs_review` status + `approve_task` / `reject_task` tools + team config flag.
**Skip:** Pattern-based review filtering (use simple boolean).

### Schema changes

- Add `needs_review` to valid TaskStatus values
- Add `review_required` field to team config (stored in teams table config JSON)

### State machine

```
pending → in_progress → needs_review → completed
                              ↓
                         in_progress (rejected with feedback)
```

When `review_required` is true in team config, `update_task(status: "completed")` transitions to `needs_review` instead.

### New MCP tools

- `approve_task`: `{ team_id, task_id, agent_id }` — lead-only, moves `needs_review → completed`
- `reject_task`: `{ team_id, task_id, agent_id, feedback }` — lead-only, moves `needs_review → in_progress` with feedback message

## Issue #3 — Git Worktree Isolation

**Core scope:** Worktree creation per teammate, cleanup on stop.

### Changes to spawn-teammate.sh

Before spawning tmux pane:
```bash
WORKTREE_DIR=".copilot-teams/worktrees/${AGENT_ID}"
BRANCH_NAME="team/${TEAM_ID}/${AGENT_ID}"
git worktree add "$WORKTREE_DIR" -b "$BRANCH_NAME"
```

Set tmux pane cwd to worktree directory.

### Changes to team-stop skill

Add worktree cleanup step:
- List worktrees matching `.copilot-teams/worktrees/*`
- `git worktree remove` each one
- Optionally delete branches

### Changes to db

Add `worktree_path TEXT` column to `members` table for tracking.

## Issue #4 — GitHub Integration

**Core scope:** `create_tasks_from_issues` tool + auto-PR on stop.
**Skip:** Issue commenting on task completion.

### New MCP tool: `create_tasks_from_issues`

Input: `{ team_id, agent_id, repo?, issue_numbers: number[] }`
- Fetches each issue via `gh issue view <n> --json title,body,labels`
- Creates one task per issue with issue metadata in description
- Stores `github_issue` number in task description for back-linking

### Auto-PR on team stop

Enhance `stop_team` tool:
- If team has completed tasks, create PR via `gh pr create`
- PR title: team goal
- PR body: task results summary with links
- Only if running in a git worktree branch (pairs with #3)

### New MCP tool: `create_pr`

Input: `{ team_id, agent_id, title?, body?, base? }`
- Creates PR from current branch
- Defaults title to team goal
- Defaults body to task results summary
